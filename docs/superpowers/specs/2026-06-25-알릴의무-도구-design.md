# 알릴의무 작성 보조 도구 — 디자인 설계

> 참고: `C:\obsidian_hoons\Hoonseo\알릴의무 프로젝트\README.md` (컨셉/룰엔진/데이터모델 원본)

## 0. 배경

표준알릴의무 5문항(Q1~Q5) 작성을 보조하는 독립 도구. 병력을 자유 텍스트로
입력하면 표로 파싱되고, 사람이 검토/수정한 뒤 룰엔진이 Q1~Q5에 분류,
체크박스로 최종 확정 후 복사한다. 자동화(날짜계산/ICD매핑/문항배치)와
사람 확인(파싱검증/확인필요 판단/최종체크)을 명확히 분리하는 것이 핵심
설계 원칙이며(README 1장), 이 문서는 그 룰엔진을 감싸는 **화면 디자인**만
다룬다. 룰엔진/파서 로직 자체는 README 2~7장이 기준.

신규 위치: `C:\Users\chlgn\OneDrive\Desktop\가입제안서PJ\알릴의무 프로젝트\`
(기존 가입제안서PJ와 별개 폴더로 시작, 추후 메인 사이트에 리다이렉트로 합류)

## 1. 전체 구조

- 독립 단일 페이지: `index.html` / `style.css` / `data.js` / `app.js`
  (README 3장 파일 구성 그대로, 경로만 하위 폴더)
- 스타일 톤은 가입제안서PJ의 `insight.html`을 계승한다 — 흰 배경,
  sticky nav, `Outfit + Noto Sans KR` 폰트, 본문 `max-width: 860px`의
  여백 넓은 에디토리얼 레이아웃. 메인 `index.html`(GA 암보장 분석기)의
  Tailwind 카드/업로드존 톤은 사용하지 않는다 — 그 쪽은 "분석 결과 감상"
  용 화려한 톤이고, 이 도구는 입력폼+편집표+체크박스 중심의 "작업 UI"라
  insight.html의 차분한 톤이 더 맞는다.
- 컬러: 보험사 무관 단일 네이비(`#1A3A8F` / `#071440`, style.css 기본
  팔레트). README에 언급된 "삼성화재 톤(#1B3A6B)"은 적용하지 않는다 —
  알릴의무 5문항은 보험사 공통 규정이라 특정 보험사 색에 종속시키지 않고,
  추후 가입제안서PJ에 합칠 때도 기본 테마로 자연스럽게 흡수되게 한다.
- 페이지는 세로 스크롤로 3단계가 순서대로 쌓인다: **입력 → 병력표 →
  분류결과**. 탭/스텝퍼 방식은 쓰지 않는다 — 작업 흐름이 순차적이고
  앞 단계 내용을 보며 뒤 단계를 확인하는 일이 잦기 때문.
- 상단 nav: 좌측 로고/타이틀, 우측 "← 가입제안서PJ로 돌아가기" 링크
  (insight.html의 `nav-back` 스타일). 이 링크가 추후 합칠 때 리다이렉트
  연결점이 된다.

## 2. Hero

insight.html의 hero 패턴을 그대로 따른다:
`hero-tag`("표준알릴의무 작성 보조") → `hero-title`("알릴의무, 자동으로
분류해보세요") → `hero-sub`(보조 도구이며 최종 확인 필수라는 면책 문구,
README 상단 인용구 톤 차용) → `hero-badges`(Q1~Q5 트리거 요약, "11대
질병" 등 태그 칩).

면책 문구는 `ce-callout warning` 스타일 박스로 한 번 더 강조한다:
"자동분류 결과는 항상 초안입니다. 최종 청약서 기재 전 직접 확인하세요."

## 3. 입력 섹션 (Step 1)

- `eyebrow`: "STEP 1 · 병력 입력" / `h2.stitle`: "병력을 자유롭게
  적어주세요"
- 큰 `textarea`, 배경 `#fafafa` / `border: 1px solid #efefef` /
  `border-radius: 16px` (insight.html `vcard` 톤). placeholder에
  README 형식 예시(`🔴진단명 .../코드`, `입원:`, `수술:`,
  `치료/현상태:`) 안내.
- 단일 CTA 버튼 "표로 변환"(네이비 필 버튼) — 클릭 시 app.js 파서가
  실행되어 결과가 Step 2 표에 행으로 추가된다. 모달 없이 같은 페이지
  스크롤 흐름.

## 4. 병력표 섹션 (Step 2)

- `eyebrow`: "STEP 2 · 병력 확인 및 수정" / `slead`: "자동 인식 결과입니다.
  날짜·일수는 직접 확인해주세요."
- `hier-box`와 같은 카드(연한 회색 배경, radius 16) 안에, 각 병력 1건이
  `coverage-entry`처럼 구분선으로 나뉜 **편집 가능한 카드형 리스트**로
  표시된다 (촘촘한 grid table 대신 — 모바일 대응 및 필드별 라벨 노출
  용이).
- 각 카드 필드(README 4장 데이터 모델 1:1 대응): 진단명/진단코드,
  최초진단일, 최근진료일, 입원여부·입원일수, 수술여부·수술명,
  계속치료일수, 계속투약일수, 재검사여부(체크), 상시복용여부(체크),
  현재상태, 비고.
- 날짜·일수 필드가 비어있으면 해당 카드 우측에 `ce-badge` 스타일의
  작은 "확인필요" 라벨이 자동으로 붙는다 (README 6장 원칙: 정보 부족 시
  무조건 노출, 자동 INCLUDE 금지). 이 라벨은 Step 2에서만 보이는 입력
  검증용이며 Step 3의 "확인필요" 배지와는 별개 — 분류 단계에서 다시
  판정한다.
- 각 카드에 행 추가(+) / 삭제 버튼.

## 5. 분류결과 섹션 (Step 3)

- `eyebrow`: "STEP 3 · 표준알릴의무 분류결과"
- Q1~Q5를 세로로 나열한 리스트. 각 문항은 `coverage-entry` 스타일
  블록: 문항 제목(`h3.ce-title`, 예: "Q1. 최근 3개월 진찰·검사") + 트리거
  한 줄 설명(`ce-oneliner`, README 2장 표 내용) + 해당 항목 체크박스
  리스트.
- 배지 색 구분: 정상 분류 항목은 `cat-s`(블루 계열), "확인필요" 항목은
  `cat-r`(주황 계열).
- 하나의 병력이 여러 문항에 동시 배치되는 경우(README 2장 예시: 고혈압
  약 상시복용 → Q2+Q4+Q5) 해당 항목명이 각 문항 줄에 중복으로 나타난다.
  항목명 클릭 시 다른 등장 위치를 하이라이트하는 기능은 1차 범위에서
  제외(추후 검토).
- 섹션 최하단에 sticky "결과 복사" 버튼 — 클릭 시 체크된 항목만 정리한
  텍스트를 클립보드에 복사하고 `toast` 스타일로 확인 노출.

## 6. 상태 / 엣지케이스

- 빈 입력으로 "표로 변환" 클릭: 토스트 "인식된 병력이 없습니다. 직접
  행을 추가해주세요" 표시 + Step 2에 빈 카드 1개 자동 생성.
- "~"로 끝나는 미완결 날짜 등 모호한 파싱 케이스: README 5장에서 발견된
  버그(자동으로 "현재진행중"/오늘 날짜로 채워 완치 사례를 "3개월 이내"로
  오분류)를 재현하지 않기 위해, 모호하면 빈 칸으로 두고 "확인필요"
  배지를 붙인다. 자동 보정 금지.
- localStorage에 입력/표 상태 자동 저장(새로고침 대비). nav에 "전체
  초기화" 버튼 — 클릭 시 확인 다이얼로그 1회 거쳐 개인정보 잔존 방지
  (README 7-4장).
- 모바일(`max-width: 640px`): Step 2 카드 내부 필드 grid 2열 → 1열로
  축소. 가입제안서PJ 기존 미디어쿼리 패턴을 따른다.

## 7. 파일 구성

```
알릴의무 프로젝트/
  index.html   nav + hero + Step1~3 섹션 마크업
  style.css    insight.html 톤 차용, 네이비 단일 팔레트, 카드/배지/토스트
  data.js      11대 질병 ICD 매핑, Q1~Q5 상수, 트리거 텍스트
  app.js       파서 + 룰엔진(분류) + 렌더링/이벤트 + localStorage
```

`data.js` 분리 이유는 README 3장과 동일: 표준사업방법서 개정으로
11대질병 범위/ICD 매핑이 바뀌어도 `app.js` 로직을 건드리지 않고 이
파일만 갱신.

추후 가입제안서PJ로 합칠 때는 이 폴더를 서브패스(예:
`/알릴의무/`)로 그대로 옮기고, 메인 `index.html`에 진입 카드 1개를
추가하고 이 도구의 nav "돌아가기" 링크를 메인 경로로 바꾸는 것만으로
통합되는 구조를 목표로 한다.
